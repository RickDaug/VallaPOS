## Top risks

1. MANAGER→OWNER privilege escalation (S1): three employee-management server actions gate only on MANAGER while the Zod roleSchema accepts OWNER and ROLE_RANK is never checked — a lesser insider or compromised MANAGER session can seize full OWNER control of a tenant (payouts, capabilities, lock out the real owner).
2. Partial-refund double-refund (S1): the client mints a fresh idempotency key per invocation, so a manual retry after a lost response — the exact intermittent-503 pattern recon observed live — can issue a second real reversing payment. Real money loss.
3. App-wide WCAG 1.4.3 contrast failure (S1): default primary (3.98:1) and success (3.62:1) Button variants render text below the 4.5:1 AA floor in light mode, affecting essentially every CTA and form submit in the product.
4. Legal exposure in production (S1): live Privacy/DMCA/Do-Not-Sell pages are self-labeled unfinished templates with unfilled placeholders, and the DMCA doc admits no registered §512 agent / no safe harbor while the product allows merchant image uploads.
5. Live register 503s / unverified DB pooling (S1): PR #115's Neon pooler fix is in source but prod still shows intermittent 503-then-retry on /register, consistent with the manual Vercel DATABASE_URL pooler swap never being applied — cashiers can randomly fail mid-sale under burst.

## Prioritized action plan

**#1 · S1 · Security & Privacy** — Add role-rank enforcement to block MANAGER→OWNER escalation
  - Evidence: src/features/employees/actions.ts: changeMemberRole (line 156) gates only assertRole(ctx,"MANAGER") at 159; addMember (61-88, gated MANAGER at 66) and addStaffMember (95-113, gated at 98) same. changeRoleSchema (schema.ts:68-73) uses roleSchema=z.enum(["OWNER","MANAGER","CASHIER"]) — OWNER is an accepted target; ROLE_RANK (roles.ts:7-11) is never consulted. Contrast setMemberPermissions (133-149) which IS OWNER-gated. A MANAGER can set/create role:"OWNER" → full tenant takeover.
  - Fix: In all three actions require assertRole(ctx,"OWNER") whenever data.role==="OWNER", and reject granting any role above the caller's own rank (ROLE_RANK[data.role] <= ROLE_RANK[ctx.role] unless ctx.role==="OWNER"). Add a regression test asserting a MANAGER caller cannot grant/set OWNER via any of the three actions.

**#2 · S1 · Functionality & User Flows** — Make partial refunds idempotent against a client retry
  - Evidence: OrderActions.tsx:57 mints a fresh crypto.randomUUID() inside run() on every invocation, including a manual retry after the catch (66-71) re-enables the button. In orders/actions.ts the only replay guard is priorReversalResult(...,clientUuid) (244), keyed on refund:<clientUuid>. Void/full-refund are safe (status flips to terminal VOIDED/REFUNDED, SETTLED set at 132) but a PARTIAL leaves PARTIALLY_REFUNDED (not terminal); a second call with a NEW uuid is blocked by neither guard, and the FOR UPDATE lock only serializes concurrent (not sequential) retries. Recon item 9 observed the live lost-response pattern.
  - Fix: Generate the clientUuid once when the partial-refund confirm dialog opens (not inside run()), or derive the server idempotency key from durable inputs (businessId+orderId+amountCents+time-bucket), or require the client to reconcile against the prior attempt's stored result before a second submit. Add a regression test for the sequential-retry case.

**#3 · S1 · Front-End, UX, Visual & Content** — Fix default Button primary/success text contrast to WCAG 1.4.3 AA
  - Evidence: app/globals.css:22-23 --primary oklch(0.58 0.1 195) on --primary-foreground oklch(0.99 0.01 200) = 3.98:1; :35-36 --success oklch(0.6 0.13 155) on --success-foreground = 3.62:1 (recomputed via independent OKLCH→sRGB→luminance, white verified). button.tsx:24 makes 'primary' the cva default; ~123 <Button> usages default to it incl. sign-up 'Create account'. Text is 'normal' size so the 3:1 large-text carve-out does not apply. Dark mode passes (7.38/7.11) — light-mode only.
  - Fix: Lower --primary OKLCH L to ~0.50-0.52 (or lighten --primary-foreground) until the pair clears 4.5:1; apply the same to --success/--success-foreground. Scope to :root so dark mode is unaffected. Re-run the contrast script to confirm both pairs >=4.5:1.

**#4 · S1 · Performance, Reliability & SEO / Back-End (shared)** — Verify (and if needed apply) the production Neon pooler DATABASE_URL swap
  - Evidence: schema.prisma:15-26 has the pooled url + directUrl split (PR #115 code present), but STATE.md notes the fix is inert until an operator swaps Vercel Prod/Preview DATABASE_URL to the -pooler/pgbouncer=true host and redeploys. Recon item 9 (live 2026-07-21) shows intermittent 503-then-200-on-retry on /{businessId}/register RSC/POST — the connection-exhaustion signature. verified:false (no prod env access this pass).
  - Fix: Via Vercel CLI/dashboard confirm Production DATABASE_URL host contains '-pooler' with pgbouncer=true and DIRECT_URL is set separately for migrations; if not, apply the env swap + redeploy, then re-run prisma/smoke-order-race.ts against prod-shaped load to confirm the 503s stop. Also wrap the Upstash calls in src/lib/redis.ts (26-34) in try/catch, a likely co-contributor.

**#5 · S1 · Front-End / Security (merged)** — Finalize live legal pages: strip template disclaimers, fill placeholders, register DMCA agent
  - Evidence: marketing-content.ts:952 live Privacy doc states "This Privacy Statement is a template... not legal advice"; :1140 DMCA doc states "...safe-harbor protections of 17 U.S.C. §512 are unavailable until a Designated Agent is registered"; unfilled [mailing address] (1000,1072,1108,1136,1162), [request form link] (1133), [DMCA Designated Agent name] (1160). ## Executive white paper

## VallaPOS White-Paper Audit — Chief Auditor's Consolidated Report

**What was audited.** VallaPOS is a browser/PWA multi-tenant point-of-sale for mobile and local businesses (food trucks, barbers, market vendors, small shops), running on Next.js 15 + React 19 + Prisma 6.9 + Better Auth on Vercel/Neon/Upstash, live at vallapos.com. The audit covered the full product surface: the money paths (checkout, refund/void, cash drawer, time tracking) and their transactional correctness; the newcomer/navigation surfaces (sign-up, onboarding, routing, error/404 boundaries, legal pages); the design system, WCAG color contrast, keyboard/touch operability, ARIA and content/copy; the back-end (server actions, webhooks, payments capture/idempotency, tenant isolation, env/DB/pooling config, Redis/auth degradation, dependency and lint policy); security and privacy (authorization/role hierarchy, tenant-isolation choke points, auth/session/webhook config, CSP and security headers, and the entire privacy/legal surface); and performance, reliability and SEO (render path, bundle weight, caching, DB indexing/pooling, metadata and structured data). Dormant, flag-gated payments (Stripe Connect, processor-QR, SaaS billing) were reviewed in source but are not live and were not exercised.

**Methodology — a three-tier audit.** Fifteen specialist auditors worked in five squads, each squad consolidated by a domain manager, and all five manager reports adjudicated here by the Chief Auditor. The audit was white-box against the source of truth at `C:/Users/RickD/AndroidStudioProjects/VallaPOS`, grounded by a single live production reconnaissance pass performed by the Audit Director (RECON.md, 2026-07-21) — the auditor agents themselves had no browser. Every surviving finding was re-read at its cited `file:line` by a manager before sign-off; no auditor claim was passed through on trust, and several severities were corrected on review (e.g. the 404 finding S3→S2, the forced-dynamic homepage S1→S2, the desktop-checkout logging gap S2→S3, and the split S1/S2 CSP finding reconciled to S2). Coverage limits are real and disclosed: live-browser recon was Director-only, so visual, a11y and rendered-pixel judgments are source-derived (contrast ratios were recomputed independently from the OKLCH tokens, which is a gamut-independent fact, but no axe/Lighthouse/screen-reader pass was run); there was no staging DB, so concurrency races and the two live-observed items (intermittent 503s, CSP header) rest on recon plus code reasoning, not re-triggered load.

**Overall health: 78/100 — conditional ship.** VallaPOS is, at the engineering level, a genuinely disciplined product — the back-end (85) and the design-system foundation are better than most pre-launch codebases, there are zero S0 defects, and there is no active cross-tenant data leak. But five S1 defects that either ship today or are live-observed keep it from an unconditional pass. Three are code-fixable in hours; one needs a ten-minute production-env check; one needs counsel plus a $6 filing.

**The top risks a stakeholder should read.** First, a **privilege-escalation hole**: any MANAGER-rank account can grant itself or an accomplice the OWNER role, because `changeMemberRole`/`addMember`/`addStaffMember` gate only on `assertRole(ctx,"MANAGER")` while the Zod schema accepts "OWNER" as a target and `ROLE_RANK` is never consulted — a straight path to full tenant takeover (payout/Stripe settings, capability changes, locking out the real owner). Second, a **double-refund money bug**: partial refunds mint a fresh idempotency key on every click, so a manual retry after a lost/ambiguous HTTP response — exactly the intermittent-503 pattern recon captured live — can issue a second real reversing payment. Third, an **app-wide accessibility failure**: the default primary and success Button variants render text below the 4.5:1 AA floor in light mode, on essentially every CTA in the product. Fourth, **legal exposure shipped to production**: the live Privacy and DMCA pages label themselves unfinished templates, carry unfilled `[mailing address]`/`[DMCA agent]` placeholders, and the DMCA doc itself admits VallaPOS has no §512 safe harbor while running a product that lets merchants upload images. Fifth, a **live reliability symptom**: production still shows intermittent 503-then-retry on the register, matching the Neon connection ceiling that PR #115's pooler fix targets — suggesting the manual Vercel `DATABASE_URL` pooler swap was never applied.

**The story.** This is not a rescue job; it is a strong codebase with a short, sharp punch-list. The build quality — compare-and-set payment capture, raw-body webhook verification, a tenant-isolation CI guard, scrypt PIN hashing, a clean OKLCH token layer — is the work of a team that knows what it is doing. What remains are a handful of concentrated, high-leverage defects: one authorization check, one idempotency-key lifetime, two color tokens, one env-var verification, and one legal-content pass. Fix those five and VallaPOS is a credible ship; below them sits a well-understood backlog of S2 reliability/perf/privacy hardening (pre-auth error boundaries, a real 404, the Payment composite index, the forced-dynamic homepage, LGPD coverage, consent-at-collection) that can follow in the normal course rather than gate launch.
