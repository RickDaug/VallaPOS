# Security & Privacy

**Manager:** Bianca, Security Lead
**Severity counts:** S0 0 · S1 1 · S2 4 · S3 3

## Executive summary

The Security & Privacy squad's most serious finding is a real, exploitable privilege-escalation defect (S1): any MANAGER-rank account can grant itself or an accomplice the OWNER role because changeMemberRole, addMember, and addStaffMember gate only on assertRole(ctx, "MANAGER") while the Zod roleSchema accepts "OWNER" as a target value — there is no check that the granted role does not exceed the caller's own rank. This directly defeats the app's documented role hierarchy and bypasses the OWNER-only capability control (setMemberPermissions) whose own code comment claims to prevent exactly this. Everything else is medium/low: a genuine config/documentation drift where CSP ships report-only in production while STATE.md and a next.config.ts comment falsely claim it is enforced (defense-in-depth layer inert, no active exploit), plus a cluster of pre-launch privacy/legal gaps — unfilled legal-notice placeholders shipped live, a US/CCPA-only privacy policy despite Brazil (LGPD) and Mexico being first-class selectable markets, no account/personal-data deletion path in code despite the policy promising one, no privacy/terms link or consent at the point of PII collection, and no email verification at sign-up. Positive controls verified as sound: tenant isolation on the reviewed queries, PIN scrypt hashing + throttling, constant-time PIN responses, raw-body Stripe webhook verification, and self-targeting clock/operator actions that derive membership from the tenant context rather than trusting client IDs.

## Findings (8)

#### S1 — Any MANAGER can self-promote (or promote an accomplice) to OWNER — no role-rank check on role grants
- **Area:** authz / privilege-escalation
- **Auditor:** Cy, the Pen Tester · **Confidence:** high
- **Evidence:** `Verified in source: src/features/employees/actions.ts changeMemberRole (line 156) gates ONLY with assertRole(ctx, "MANAGER") at line 159; the only additional guard (lines 167-172) blocks demoting the last OWNER, nothing blocks setting role:"OWNER". data.role comes from changeRoleSchema (schema.ts:68-73) using roleSchema = z.enum(["OWNER","MANAGER","CASHIER"]) (schema.ts:22) — OWNER is an accepted target. Identical unrestricted pattern in addMember (actions.ts:61-88, gated MANAGER at line 66) and addStaffMember (actions.ts:95-113, gated MANAGER at line 98): a MANAGER can create a brand-new OWNER outright. ROLE_RANK (src/lib/roles.ts:7-11) CASHIER=0/MANAGER=1/OWNER=2 is never consulted to cap the granted role. Contrast setMemberPermissions (actions.ts:133-149) which IS OWNER-gated (line 136) with a comment stating the intent to stop a manager escalating access — an intent this bug bypasses.`
- **Impact:** A MANAGER-level account (a legitimate lesser insider, or a compromised MANAGER session/PIN) can grant itself or a planted accomplice full OWNER rights within the tenant, then use OWNER-only actions to change payout/Stripe Connect settings, alter capabilities, and demote or lock out the real owner — full takeover of the business. Defeats the documented role hierarchy and the explicit control setMemberPermissions was built to provide.
- **Fix:** In changeMemberRole, addMember, and addStaffMember require assertRole(ctx, "OWNER") whenever data.role === "OWNER", and ideally reject granting any role above the caller's own rank (ROLE_RANK[data.role] <= ROLE_RANK[ctx.role] unless ctx is OWNER). Add a regression test asserting a MANAGER caller cannot grant/set OWNER via any of the three actions.

#### S2 — CSP ships report-only in production while STATE.md and a next.config.ts comment claim it is enforced (documentation drift)
- **Area:** xss / config-drift
- **Auditor:** Cy, the Pen Tester + Rosa, the Config Auditor (deduped) · **Confidence:** high
- **Evidence:** `Verified in source: middleware.ts:43 const CSP_ENFORCE = false; and CSP_HEADER (lines 45-47) resolves to Content-Security-Policy-Report-Only whenever the flag is false — matching RECON.md's live prod header capture. The in-file comment (lines 24-42) documents this as a deliberate choice pending live-device verification. However next.config.ts:4 comments the CSP is "now ENFORCED (R-5)" and STATE.md's 2026-06-25 #77 entry claims CSP was "moved from report-only to ENFORCED with per-request nonces... Verified live on prod." docs/AUDIT_EVIDENCE.md correctly says report-only. Code + one doc agree with prod; STATE.md and the next.config.ts comment assert the opposite. Severity reconciled from Rosa's S1 / Cy's S2 to S2: both auditors agree nothing is actually blocked and there is no active exploit — the risk is an inert defense layer plus misleading posture docs, not a live vulnerability.`
- **Impact:** No live exploit by itself (report-only blocks nothing, only reports). The harm is that a future engineer, auditor, or incident responder relying on STATE.md/next.config.ts would believe script-injection XSS is mitigated in production when it is not, potentially deprioritizing a real fix and running one security layer thinner than assumed.
- **Fix:** Correct STATE.md #77 and the next.config.ts comment to state CSP remains report-only pending live-device verification (matching middleware.ts and docs/AUDIT_EVIDENCE.md). Separately, since Upstash rate limiting is live and the E2E harness runs against prod, schedule the live-device verification pass so CSP_ENFORCE can be flipped to true for real.

#### S2 — Live privacy/legal pages ship with unfilled template placeholders
- **Area:** privacy-legal
- **Auditor:** Idris, the Privacy Officer · **Confidence:** high
- **Evidence:** `Verified via grep of src/features/marketing/marketing-content.ts: the literal "[mailing address]" appears at lines 1000, 1072, 1108, 1136, and 1162 (Privacy Statement, Terms of Use, Dispute Policy, DMCA/notice, and the Do-Not-Sell notice), and "[request form link]" at line 1133 inside the CCPA rights-request text. This file is the live marketing site's content source (shipped in PR #127, generated via scripts/marketing/generate.mjs), not a draft fixture.`
- **Impact:** The CCPA/CPRA notice and Terms of Use are legally-facing documents that promise a mailing address and a rights-request form; shipping bracketed placeholders undermines their validity as the required notices (CCPA/CPRA expects an accessible means to submit requests; ToS is expected to carry a real registered-agent/address). A user or regulator sees an obviously unfinished legal document.
- **Fix:** Before public launch, fill in a real mailing/registered-agent address and either build the referenced rights-request form or remove the dangling reference and rely on the already-wired privacy@/legal@ addresses.

#### S2 — Privacy policy addresses only US/CCPA law despite Brazil (LGPD) and Mexico being supported, selectable markets
- **Area:** privacy-legal
- **Auditor:** Idris, the Privacy Officer · **Confidence:** high
- **Evidence:** `Verified: the Privacy Statement (marketing-content.ts:987-1000) and Do-Not-Sell notice (1110-1138) reference only "California and other US states with privacy laws" / CCPA-CPRA; a repo-wide grep for LGPD and GDPR across src/features, src/lib, and app returned zero hits (only the confirmation there is no non-US privacy content anywhere). RECON.md item 6 plus src/features/auth/actions.ts COUNTRY_CODES confirm BR/MX/CA are first-class selectable business countries, so a Brazilian merchant (an LGPD data subject) is an explicitly supported user with rights the policy never addresses.`
- **Impact:** A Brazilian merchant the product explicitly onboards has LGPD rights and VallaPOS (as processor for that merchant's customer data, per the policy's own controller/processor framing) has undocumented obligations. Real compliance gap for a live, selectable, non-US market — not a hypothetical EU visitor.
- **Fix:** Add an LGPD section (mirroring the existing CCPA one) naming the legal basis for processing and the Brazilian data-subject-rights process; have counsel confirm GDPR exposure for any EU visitors before scaling marketing spend into those regions.

#### S2 — No self-service or automated account/personal-data deletion path in code despite the policy promising one
- **Area:** pii-retention
- **Auditor:** Idris, the Privacy Officer · **Confidence:** medium
- **Evidence:** `Verified: the Privacy Statement promises deletion on request (marketing-content.ts:979) and the Do-Not-Sell notice lists "Delete" as a right (line 1125). A repo-wide grep for deleteAccount|deleteUser|right to be forgotten|LGPD|GDPR across src/features, src/lib, and app returned zero implementing code — only policy prose. src/features/auth/actions.ts (createBusiness/getPrimaryBusinessId) has no delete/export counterpart and no admin tooling for it was found.`
- **Impact:** Deletion is apparently a fully manual, undocumented-in-code process triggered by emailing privacy@vallapos.com. With no code path there is no way to verify a request is honored completely (cascading Business/Order/Membership/User rows, offline-queue snapshots, Resend delivery logs) or within the CCPA/LGPD statutory timeframe, and no evidence the process exists outside the promise text.
- **Fix:** Build (or explicitly document as an operational runbook if intentionally manual for now) a deletion procedure covering all PII-bearing tables (User, Membership staff name, Business, Order/Payment records with customer receipt emails) and log fulfillment for audit.

#### S3 — Sign-up form collects PII with no link to Privacy Policy or Terms of Use and no consent notice
- **Area:** pii-consent
- **Auditor:** Idris, the Privacy Officer · **Confidence:** high
- **Evidence:** `Verified: a grep for terms|privacy|policy across app/(auth)/sign-up/page.tsx returned zero hits. RECON.md item 6 confirms the field list (name, business name, business type, country, email, confirm email, password) with no legal-document reference. The Privacy Statement and Terms are only reachable from the marketing site footer (marketing-content.ts:941,944), not from the auth flow.`
- **Impact:** A new user hands over name/email/business data with no visible disclosure of, or link to, the privacy policy or terms at the point of collection — weaker than standard "by signing up you agree to our Terms and Privacy Policy" practice and a soft gap against CCPA's notice-at-collection expectation.
- **Fix:** Add a "By creating an account you agree to our Terms and Privacy Policy" line with links under the sign-up form.

#### S3 — No email verification at sign-up — an unverified address becomes the receiver of account communications
- **Area:** auth / pii-integrity
- **Auditor:** Idris, the Privacy Officer · **Confidence:** high
- **Evidence:** `Verified in source: src/lib/auth.ts emailAndPassword (lines 42-54) enables password auth and wires sendResetPassword but declares no emailVerification / requireEmailVerification config. RECON.md item 6 independently confirms signup logs straight into /{businessId}/register with no verification wall.`
- **Impact:** An unverified email means VallaPOS cannot confirm the stored account belongs to a real, consenting owner of that address — password-reset links, receipts, and account notices could be sent to a mistyped or someone else's address with no confirmation loop (account-integrity and minor account-takeover surface).
- **Fix:** Add Better Auth email verification (even soft/non-blocking) so the primary email is confirmed before account communications rely on it.

#### S3 — CSP violation report endpoint (/api/csp-report) has no rate limiting
- **Area:** headers/CSP
- **Auditor:** Rosa, the Config Auditor · **Confidence:** medium
- **Evidence:** `Verified per RECON + source: app/api/csp-report/route.ts's POST handler is unauthenticated by CSP design, capped at a 16KB body (MAX_BYTES) and always returns 204, but has no per-IP/per-tenant throttle (unlike the Upstash-backed auth endpoints). It never reflects or executes input. This is the practical downstream of the CSP-report-only finding: while CSP_ENFORCE is false, injected script still runs in the browser and this collector only records.`
- **Impact:** Low on its own — no reflection or execution, so a flood is a log-noise / minor-cost nuisance rather than an exploit. Noted mainly as the practical consequence of CSP being observational rather than a working XSS block today.
- **Fix:** Optional: add a coarse IP-based rate limit to /api/csp-report once Upstash reachability from the Edge route is confirmed, purely to bound log volume. The primary fix remains resolving the CSP report-only / doc-drift item.
